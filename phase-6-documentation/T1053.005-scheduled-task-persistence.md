### What Was Built

A fully instrumented home SOC lab environment simulating a realistic enterprise
security operations workflow against a CIS-hardened Windows Server 2022 Domain
Controller. The lab implements every component of the detection-to-remediation
chain independently and validates each link in that chain against live adversary
simulation using Atomic Red Team.

The capstone is not a tutorial replication. Every detection rule was built from
scratch against real telemetry. Every false positive was encountered, documented,
and excluded with evidence. Every audit gap was found, traced to its root cause,
and compensated for. The infrastructure troubleshooting was real — including a
Python version conflict requiring a virtual environment to resolve
ansible-core compatibility with community.windows 3.x.

## Key Findings Across Completed Investigations

The following findings represent the most technically significant discoveries
across all completed investigations — things that were not obvious going in
and required actual investigation to surface.

**Finding 1 — Port 9389 ADWS fingerprint (INV-003)**  
PowerShell AD cmdlets (Get-ADUser, Get-AdGroup, Get-ADObject) communicate over
port 9389 — not the traditional LDAP ports 389/636. Detection rules monitoring
only LDAP ports miss PowerShell-based AD enumeration entirely. The correct
detection requires a Sysmon Event ID 22 (DNS query of DC FQDN) immediately
preceding Sysmon Event ID 3 (port 9389 connection). This finding is not
widely documented in SOC detection literature.

**Finding 2 — 5140 user vs machine account (INV-004)**  
Windows generates 5140 (Network Share Object Accessed) for both machine accounts
(background OS processes) and user accounts (human or attacker activity).
Machine account events (WIN-hostname$) are baseline noise. User account events
(no $ suffix) accessing administrative shares (C$, ADMIN$, IPC$) are a
high-fidelity standalone behavioral indicator of human-initiated share
enumeration — independently detectable without process creation correlation.

**Finding 3 — TaskContent XML is the highest-value field (INV-005)**  
Event ID 4698 (Scheduled Task Created) contains a TaskContent field with the
complete XML definition of the registered task — action, trigger, principal,
and settings. For encoded execution chains, this field exposes the IEX/Base64
payload that is invisible in every other telemetry source. No process creation
event, no Sysmon event, and no registry event captures the encoded payload
directly — only the TaskContent XML does. Detection rules for T1053.005 that
do not inspect TaskContent are incomplete.

**Finding 4 — Tarrask detection via SD absence (INV-005)**  
The Windows Task Scheduler writes a Security Descriptor (SD) registry value
under HKLM\...\TaskCache\Tree\[TaskName]\ for every normally registered task.
The Tarrask hidden task technique deletes this SD value (requiring SYSTEM
privileges), rendering the task invisible to schtasks /query. The absence of
a Sysmon 13 SD write following a 4698 task creation event is therefore a
high-confidence indicator of the Tarrask variant — a detection signal
derivable from existing telemetry without additional tooling.

**Finding 5 — ansible.windows vs community.windows (INV-005)**  
The module win_scheduled_task does not exist in the ansible.windows collection.
It belongs to community.windows. This distinction is not prominently documented
and caused significant troubleshooting during INV-005 Ansible implementation.
Additionally, community.windows 3.x requires ansible-core 2.16+, which in turn
requires Python 3.10+ — a dependency chain not obvious from surface-level
documentation.

## Lessons Learned

### Technical Lessons

1. **Velociraptor technique-specific hunts must precede cleanup** — ART cleanup removes artifacts. The hunt must capture live state while artifacts exist. This was a persistent gap through INV-004 before being corrected in INV-005.

2. **Sysmon consistently outperforms 4688** — Sysmon 1 captures parent process chains that 4688 misses (svchost.exe → schtasks.exe in INV-005). Sysmon 1 captures inline PowerShell cmdlet execution that never generates 4688. A detection strategy relying solely on 4688 has documented blind spots.

3. **TaskContent inspection is mandatory for scheduled task detection** — Binary presence/absence of schtasks.exe is insufficient. The execution content matters. IEX/Base64 in TaskContent XML is the CRITICAL-tier indicator.

4. **Python version compatibility affects the entire toolchain** — ansible-core, collection versions, and pywinrm all have interdependencies that are not obvious until you hit them. Documenting the resolution chain saves time in future lab rebuilds.

5. **n8n Code node input reference changes after action nodes** — On branches preceded by SSH or other action nodes, `$input` contains the action output, not the webhook payload. Use `$('Webhook').first().json` to reference the original payload reliably.

### Investigative Lessons

1. **Write SPL before execution** — Pre-test SPL disciplines the analyst to think through what they expect to see before seeing it. Post-hoc SPL writing produces confirmation bias, not investigation.

2. **Bucket D findings require the same rigor as malicious findings** — Absent events that were predicted require formal root cause analysis. Saying "it wasn't there" without explaining why is not acceptable investigation practice.

3. **False positives must be documented with evidence** — VSSVC.exe, PSHost pipes, BAM writes — none of these are dismissed by pattern recognition alone. Each requires a formal disposition with field evidence.

4. **Layer 4 is the deliverable** — The narrative synthesizes everything and is what an employer reads. Every assertion in Layer 4 must be traceable to a specific log, field, and value.

### Workflow Lessons

1. **One technique per conversation** — Context management is critical for AI-assisted investigation. A fresh conversation with the master context prompt preserves analytical quality and minimizes cost.

2. **Infrastructure gates before investigation** — n8n, Ansible, and WinRM connectivity must be confirmed before opening a full vertical slice investigation. Starting the investigation with unresolved infrastructure issues delays completion and fragments context.

3. **Cleanup always runs last** — After all deliverables are complete, not before. ART artifacts are evidence until the investigation closes.

## Sigma Rule Integration — Future Work

*(Discussion held at close of INV-005 — documented here for Phase 6 planning)*

Sigma rules as currently implemented are static YAML files stored in GitHub.
They are not integrated with any active detection platform. The gap between
writing a Sigma rule and operationalizing it represents a meaningful portfolio
enhancement opportunity.

**Option 1 — Sigma → Splunk via sigma-cli:**  
The sigma-cli tool converts Sigma rules to platform-specific query language.
Converting INV-005's Sigma rule to Splunk SPL and comparing the output to
the hand-written detection rule demonstrates both Sigma authorship and
toolchain proficiency.

```bash
pip install sigma-cli
sigma convert -t splunk -p splunk_windows sigma-rule-T1053.005.yml
```

**Option 2 — Sigma → Elastic/OpenSearch:**  
Deploying a lightweight Elastic or OpenSearch instance alongside Splunk and
converting Sigma rules to Elastic Query Language demonstrates multi-platform
detection engineering — a stronger portfolio signal than single-platform coverage.

**Option 3 — Sigma rule repository integration:**  
Contributing rules to the SigmaHQ community repository after completing the
capstone provides public attribution and community validation of detection logic.

**Recommended approach:** Option 1 first — sigma-cli integration with Splunk
adds immediate value to the existing platform without new infrastructure.
Option 3 at project close for portfolio visibility.
