## Overview

Phase 5 documents the Infrastructure as Code (IaC) remediation layer
implemented using Ansible. Per-technique remediation plays are developed
alongside each full vertical slice investigation and accumulate into a
master playbook covering all simulated techniques by Phase 6.

Ansible remediation serves as the automated response component triggered
by n8n when a CRITICAL-severity detection fires. Each play is designed to
remove the specific artifacts created by the simulated technique — not
broad system changes — ensuring precise, targeted remediation that can be
safely executed in production-equivalent environments.

## Collection Dependencies

**Required collections:**

| Collection | Version | Purpose | Install command |
|------------|---------|---------|-----------------|
| community.windows | 3.1.0 | win_scheduled_task, win_regedit | ansible-galaxy collection install community.windows |
| ansible.windows | 3.5.0 | Core Windows modules | ansible-galaxy collection install ansible.windows |

**Critical note — module namespace:**  
`win_scheduled_task` and `win_regedit` belong to `community.windows`, NOT
`ansible.windows`. The module `ansible.windows.win_scheduled_task` does not
exist. Always use `community.windows.win_scheduled_task`.

**Required Python packages (in venv):**

| Package | Purpose | Install command |
|---------|---------|-----------------|
| pywinrm | WinRM connection to Windows targets | /home/merlin/ansible-venv/bin/pip install pywinrm |

**pywinrm note:** Must be installed explicitly into the venv. Not inherited
from system Python 3.9 environment. Absence of pywinrm produces error:
`winrm or requests is not installed: No module named 'winrm'`

## Infrastructure Troubleshooting Log

The following issues were encountered and resolved during initial Ansible
configuration for INV-005. Documented here to prevent recurrence across
future investigations.

**Issue 1 — ansible.windows.win_scheduled_task module not found**
- Root cause: Module belongs to community.windows, not ansible.windows
- Resolution: Replace all `ansible.windows.win_scheduled_task` references with `community.windows.win_scheduled_task`

**Issue 2 — ansible-core 2.14.18 incompatible with community.windows 3.x**
- Root cause: community.windows 3.x requires ansible-core 2.16+. Rocky Linux system Python 3.9 limits pip to ansible-core 2.15.x
- Resolution: Install Python 3.11 via dnf, create virtual environment, install ansible-core 2.19.8

**Issue 3 — Two ansible-core versions coexisting**
- Root cause: pip installed 2.15.x to ~/.local under Python 3.9; 2.19.8 installed under Python 3.11 venv
- Resolution: Auto-activate venv via ~/.bashrc — `source /home/merlin/ansible-venv/bin/activate` appended to shell profile

**Issue 4 — pywinrm not available in venv**
- Root cause: pywinrm installed at system Python level not inherited by venv
- Resolution: `/home/merlin/ansible-venv/bin/pip install pywinrm`

**Issue 5 — n8n SSH node not activating venv**
- Root cause: n8n SSH sessions are non-interactive — shell profile not sourced
- Resolution: Call venv ansible-playbook binary directly by full path:
  `/home/merlin/ansible-venv/bin/ansible-playbook`

**Chronological resolution steps:**
1. Identified module namespace error (ansible.windows vs community.windows)
2. Installed Python 3.11 via `sudo dnf install -y python3.11`
3. Created venv: `python3.11 -m venv /home/merlin/ansible-venv`
4. Installed ansible-core: `python3.11 -m pip install ansible-core`
5. Confirmed ansible-core 2.19.8 in venv
6. Appended venv activation to ~/.bashrc
7. Installed community.windows: `ansible-galaxy collection install community.windows`
8. Installed pywinrm: `/home/merlin/ansible-venv/bin/pip install pywinrm`
9. Syntax check passed: `ansible-playbook [playbook] --syntax-check`
10. n8n SSH node updated to use direct venv binary path

## T1053.005 — Scheduled Task Persistence Remediation

**File:** /home/merlin/ansible-lab/capstone1/t1053005-remediation.yml  
**GitHub:** `plays/t1053005-remediation.yml`  
**Technique:** T1053.005 — Scheduled Task/Job: Scheduled Task  
**Triggered by:** n8n CRITICAL severity branch via SSH  
**Validated:** 2026-04-19

### Artifacts Targeted

| Artifact | Type | Method | Module |
|----------|------|--------|--------|
| \T1053_005_OnLogon | Scheduled task | state=absent | community.windows.win_scheduled_task |
| \T1053_005_OnStartup | Scheduled task | state=absent | community.windows.win_scheduled_task |
| \AtomicTask | Scheduled task | state=absent | community.windows.win_scheduled_task |
| \T1053_005_WMI | Scheduled task | state=absent | community.windows.win_scheduled_task |
| \ATOMIC-T1053.005 | Scheduled task | state=absent | community.windows.win_scheduled_task |
| HKCU:\SOFTWARE\ATOMIC-T1053.005 | Registry key | state=absent | community.windows.win_regedit |

**Two-artifact remediation rationale:**  
The HKCU\SOFTWARE\ATOMIC-T1053.005 registry key stores the Base64-encoded
payload for the Qakbot-style task (Test 7). This key is invisible to the
Windows audit pipeline (4657 absent — CIS Level 1) and Sysmon 13 (path
outside monitored scope). Despite absent direct telemetry, the key's existence
is confirmed by 4698 TaskContent XML referencing it in the IEX chain.
Ansible remediation must target both artifacts to achieve complete clean-up —
removing only the scheduled task leaves the encoded payload in the registry.

### Playbook Content

```yaml
---
- name: T1053.005 Remediation - Remove Malicious Scheduled Tasks
  hosts: windows
  gather_facts: false

  vars:
    atomic_tasks:
      - "T1053_005_OnLogon"
      - "T1053_005_OnStartup"
      - "AtomicTask"
      - "T1053_005_WMI"
      - "ATOMIC-T1053.005"
    atomic_registry_key: 'HKCU:\SOFTWARE\ATOMIC-T1053.005'

  tasks:

    - name: Remove ART scheduled tasks
      community.windows.win_scheduled_task:
        name: "{{ item }}"
        state: absent
      loop: "{{ atomic_tasks }}"
      ignore_errors: true
      register: task_removal_results

    - name: Log task removal results
      ansible.builtin.debug:
        msg: "Task {{ item.item }} removal status: {{ item.changed }}"
      loop: "{{ task_removal_results.results }}"

    - name: Remove ATOMIC-T1053.005 registry key
      community.windows.win_regedit:
        path: HKCU:\SOFTWARE\ATOMIC-T1053.005
        state: absent
      ignore_errors: true
      register: registry_removal_result

    - name: Log registry removal result
      ansible.builtin.debug:
        msg: "Registry key {{ atomic_registry_key }} removal status:
              {{ registry_removal_result.changed }}"

    - name: Verify scheduled tasks are absent
      ansible.windows.win_shell: |
        $tasks = @(
          "T1053_005_OnLogon",
          "T1053_005_OnStartup",
          "AtomicTask",
          "T1053_005_WMI",
          "ATOMIC-T1053.005"
        )
        foreach ($task in $tasks) {
          $result = Get-ScheduledTask -TaskName $task `
                    -ErrorAction SilentlyContinue
          if ($result) {
            Write-Output "FAIL: $task still present"
          } else {
            Write-Output "PASS: $task absent"
          }
        }
      register: verification_output

    - name: Display verification results
      ansible.builtin.debug:
        msg: "{{ verification_output.stdout_lines }}"

    - name: Verify registry key is absent
      ansible.windows.win_shell: |
        $key = Get-ItemProperty -Path 'HKCU:\SOFTWARE\ATOMIC-T1053.005' `
               -ErrorAction SilentlyContinue
        if ($key) {
          Write-Output "FAIL: Registry key still present"
        } else {
          Write-Output "PASS: Registry key absent"
        }
      register: registry_verification

    - name: Display registry verification
      ansible.builtin.debug:
        msg: "{{ registry_verification.stdout_lines }}"
```

### Execution Notes

- `ignore_errors: true` on task and registry removal tasks allows the play
  to continue if an artifact is already absent — idempotent behavior
- Verification tasks use PowerShell via win_shell to provide explicit
  PASS/FAIL output per artifact — independent of Ansible changed/ok status
- gather_facts: false reduces connection overhead — facts not required
  for this remediation play
