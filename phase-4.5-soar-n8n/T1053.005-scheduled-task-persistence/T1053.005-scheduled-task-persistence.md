## Overview

Phase 4.5 documents the Security Orchestration, Automation, and Response
(SOAR) layer implemented using n8n. This phase enters the capstone workflow
at T1053.005 — the first full vertical slice investigation — and produces
an automated response workflow for every subsequent full slice technique.

The n8n platform bridges Splunk detection alerts and Ansible remediation,
implementing severity-based branching that determines whether a detection
event warrants automated remediation or analyst notification only. This
design reflects real-world SOAR implementation principles: not every alert
triggers automated action, and the decision threshold is encoded in the
workflow logic rather than left to manual triage.

## Splunk Integration

**Integration method:** Splunk Saved Search → Alert Action → Webhook  
**Webhook URL:** `http://10.0.20.100:5678/webhook/splunk-t1053`  
**Payload format:** JSON  
**Trigger:** Per result (each alert result sends a discrete POST)

**Splunk alert configuration:**
```
Alert name:  [Persistence] T1053.005 - Suspicious Scheduled Task Creation
Schedule:    */5 * * * * (every 5 minutes)
Expires:     24 hour(s)
Trigger:     For each result
Throttle:    5 minutes on Task_Name field
Severity:    High (static alert classification)
Actions:     Add to Triggered Alerts + Webhook
Webhook URL: http://10.0.20.100:5678/webhook/splunk-t1053
```

**Payload fields consumed by n8n:**
```json
{
  "body": {
    "result": {
      "severity": "CRITICAL|HIGH|MEDIUM",
      "Task_Name": "\\TaskName",
      "Account_Name": "MERLIN",
      "EventCode": "4698",
      "_time": "2026-04-17T23:03:32"
    }
  }
}
```

**Field path note:** n8n Code nodes referencing webhook payload on branches
preceded by action nodes must use `$('Webhook').first().json` rather than
`$input` — `$input` on those branches contains action node output, not the
original webhook payload.

### Severity Branching Logic

| Severity | Source | n8n branch | Action |
|----------|--------|-----------|--------|
| CRITICAL | 4698 TaskContent contains IEX/FromBase64String/EncodedCommand/GetItemProperty | TRUE | SSH → Ansible remediation + log |
| HIGH | 4698 TaskContent contains powershell/cmd.exe/wscript/cscript/mshta/rundll32; OR 4688/Sysmon 1 process creation events | FALSE | Log response timeline only |
| MEDIUM | 4698 without HIGH/CRITICAL pattern | FALSE | Log response timeline only |

## Infrastructure Notes

**n8n provisioning decisions:**

The decision to deploy n8n on a dedicated Docker VM (10.0.20.100) rather than
the Splunk/Velociraptor Ubuntu server (10.0.10.20) was made after resource
verification. The Splunk/Velociraptor server operates with adequate headroom
for its workload, but isolating n8n on a dedicated Docker host with Portainer
already installed provides cleaner architecture and easier container management
for the lab environment.

**Network reachability verified:**
- Linux Mint (10.0.99.10) → n8n (10.0.20.100:5678) ✅
- Splunk/Velociraptor (10.0.10.20) → n8n (10.0.20.100:5678) ✅
- n8n (10.0.20.100) → Rocky Linux (10.0.10.105:22) ✅
- Rocky Linux (10.0.10.105) → Windows DC (10.0.10.10:5986) ✅

pfSense does not currently enforce VLAN segmentation — intentional for lab
ease of use. Full segmentation will be evaluated in Phase 6 documentation
as a future hardening recommendation.

**Splunk log cleanup performed 2026-04-19:**
- Removed rotated logs from /opt/splunk/var/log/splunk/
- Recovered approximately 1.4GB on Splunk/Velociraptor Ubuntu server
- Command: `sudo find /opt/splunk/var/log/splunk/ -name "*.log.*" -delete`
